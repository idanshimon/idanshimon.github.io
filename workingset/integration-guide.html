Saving tokens in your custom agent

A practical playbook for engineers building their own agent stacks who want to cut input-token cost without ripping out their existing infra. Covers two complementary approaches — Derek Thomas's ContextForge (model-server layer) and workingset (markdown-vault layer). You probably want one, the other, or both depending on what you control.

TL;DR — Which one do you need?

  • You run your own model server (vLLM, TGI, llama.cpp, your own inference stack)? → ContextForge. Operates at the KV cache. Paper benchmarks 4-13× reductions.
  • You call Anthropic / OpenAI / Copilot CLI / any hosted API? → workingset. You don't have KV cache access; this works at the markdown layer instead. 30-60× reduction on real customer-notes data.
  • Both? Stack them. workingset shrinks the payload before it leaves your agent; ContextForge optimizes how that payload hits your model. Compose cleanly.
  • Your agent doesn't load files (pure code generation, no document context)? → Neither tool helps. Look at MCP, prompt caching, or shorter system prompts instead.

Where the tokens actually go in your agent

Before integrating anything, profile your existing token spend. The cheapest optimization is the one targeting your biggest line item. A typical custom agent's per-request token breakdown looks like this:

SourceTypical sizeHow often it repeats
Harness system prompt5K-50K tokensEvery request
Tool definitions (JSON schemas)2K-20K tokensEvery request
User-provided context (docs, files, history)10K-200K tokensEvery request in a loop
Conversation historyGrows over timeRe-sent every turn
Tool call results1K-50K eachStays in history
The user's actual question~100 tokensOnce

For most agent stacks, the "user-provided context" line dominates if you load docs/files. workingset addresses this line specifically. ContextForge addresses the cost of re-processing that context on each loop iteration.

Step zero: instrument before you optimize

Add token logging to every model call in your harness. Anthropic returns usage.input_tokens + usage.output_tokens + usage.cache_read_input_tokens in every response. OpenAI same. Log them. Without numbers you'll spend a week building the wrong optimization.

Path A: integrate workingset (markdown layer)

Use this if your agent reads files from disk as context (Obsidian vault, customer-notes repo, project tracker, knowledge base, internal wiki, anything markdown-shaped).

Step 1 — install + verify there's value on your data

$ pip install git+https://github.com/idanshimon/workingset
$ cd ~/path/to/your/vault
$ ws init
$ ws diff some/branch --regenerate

Look at the ratio. Anything under 2× means workingset isn't a fit for that vault shape. 5×+ is the green zone. The real-world customer-notes vault that originated this tool benchmarks at 34×.

Step 2 — generate briefs

$ ws brief some/branch --budget 8000 --write
# Writes /brief.md (deterministic shape, ~8K tokens)

The 8K default is the floor where an agent can "work from the brief without re-reading." Smaller saves more but forces re-reads, which negate the savings.

Step 3 — swap your "load context" function

The integration is literally one function swap. The pattern across most custom agent stacks looks something like:

Before — naive load 
def load_customer(slug: str) -> str:
    """Load all source files for a customer branch."""
    base = Path(f"~/vault/cust/{slug}").expanduser()
    parts = []
    for f in sorted((base / "context").glob("*.md")):
        parts.append(f.read_text())
    return "\n\n---\n\n".join(parts)
    # ~57,000 input tokens per load on a real vault
After — brief-first 
def load_customer(slug: str) -> str:
    """Load the workingset brief; fall back to source on demand."""
    brief = Path(f"~/vault/cust/{slug}/brief.md").expanduser()
    if brief.exists():
        return brief.read_text()
    # Cold path: vault hasn't been workingset-indexed yet
    import subprocess
    subprocess.run(["ws", "brief", f"cust/{slug}", "--budget", "8000", "--write"])
    return brief.read_text()
    # ~1,700 input tokens per load — same model, same harness

Step 4 — fall-back to source when needed

The brief covers ~80% of "load context" cases. The remaining 20% are when the user asks for verbatim quotes, deep historical context, or something the brief explicitly says "not in this context" for. Wire a tool that lets the agent escalate:

def load_customer_source_file(slug: str, filename: str) -> str:
    """Tool the agent can call when brief is insufficient."""
    p = Path(f"~/vault/cust/{slug}/context/{filename}").expanduser()
    if not p.exists():
        return f"File not found: {filename}"
    return p.read_text()

In your tool definitions, mark this as a fall-back: "Use only when the brief returns 'not in this context' or the user explicitly asks for verbatim historical content."

Step 5 — automate refresh

$ cat > ~/.local/bin/refresh-briefs.sh << 'EOF'
#!/bin/bash
set -e
WS=$(which ws)
cd ~/path/to/your/vault
"$WS" reindex
for branch in cust/*; do
    [ -d "$branch" ] && "$WS" brief "$branch" --budget 8000 --write
done
EOF
$ chmod +x ~/.local/bin/refresh-briefs.sh
$ (crontab -l; echo "0 4 * * *  ~/.local/bin/refresh-briefs.sh > /tmp/ws.log 2>&1") | crontab -

4am daily is a sane default. The brief stays useful for ~24 hours unless your vault is extremely volatile.

Safety note — only matters with small models

If your custom agent routes through a frontier model (Claude Opus / Sonnet, GPT-5+, Gemini 2.5+), the brief is verified safe — 0% hallucination on out-of-brief questions across 432 benchmark trials. If you route through Haiku / GPT-5.4-mini / similar, expect ~33% hallucination rate on questions whose answer isn't in the brief. Either use a frontier model for safety-critical loads OR add the source-file-fallback tool from Step 4 and instruct the agent to use it whenever certainty matters.

Path B: integrate ContextForge (model-server layer)

Use this if you control the model serving layer (running vLLM, TGI, your own llama.cpp deployment, etc) OR if you're using a provider that exposes KV cache control (very few do; check their docs).

ContextForge's contribution is not the file-shrinking step — it's the reuse of the model's already-computed KV cache state when subsequent requests share a prefix. The paper benchmarks: 4-13× reduction in compute on long-context tasks vs naive recomputation.

What you need from ContextForge

The 5-layer architecture from Derek's paper:

LayerWhat it isYou need it if...
L−1 LoRAFine-tuning the model on your domainYou're willing to maintain a LoRA adapter; orthogonal to context
L0 ResidualPre-computed compact summary (the "brief")Same intent as workingset — pre-compute the context
L1 Branch cacheActive branch's KV cache stays warm in GPU memoryYou serve requests at scale and want to skip re-prefill
L2 Memory indexBM25 retrieval over the corpusYou need fast lookup beyond what the brief covers
L3 Knowledge storeThe raw filesAlways — this is your source of truth

Step 1 — clone + read the architecture

$ git clone https://github.com/Betanu701/ContextForge
$ cd ContextForge
$ cat docs/architecture.md   # Derek's full walkthrough

Step 2 — figure out which layers you actually need

You don't have to adopt all 5. The minimum useful subset for token reduction:

  • L0 (residual brief) — same idea as workingset's brief.md
  • L1 (branch cache) — keep the KV cache for active branches warm in GPU memory so successive requests skip prefill

L2 (index) is helpful if your queries don't all hit the brief. L−1 (LoRA) is its own project. L3 (store) is just "your files exist."

Step 3 — wire L1 branch cache into your model server

This is the tricky part and the part workingset can't do for you. The interface looks like:

# Pseudocode — adapt to your model server
from contextforge import BranchCache, BriefStore

cache = BranchCache(max_branches=20, eviction="lru")
briefs = BriefStore(path="./briefs")

def handle_request(branch: str, query: str):
    # Check if we have a warm KV cache for this branch
    if branch in cache:
        kv_state = cache.get(branch)
        # Resume from cached state — skip prefilling the brief
        response = model.generate(query, resume_from=kv_state)
    else:
        # Cold path: prefill the brief, cache the resulting KV
        brief = briefs.load(branch)
        kv_state, response = model.generate_and_cache(brief + query)
        cache.put(branch, kv_state)
    return response

Your model server has to expose "generate from this KV state" as an API. vLLM supports this via --enable-prefix-caching. TGI has similar. Closed APIs do not.

Step 4 — measure

The win shows up as GPU compute reduction, not directly as tokens. Track:

  • Prefill time — should drop dramatically on warm branches (10x+ for long prefixes)
  • Time-to-first-token — same direction, user-visible
  • GPU memory pressure — branch cache uses memory; tune max_branches

Path C: stack both (best of both)

If you control the model server AND you have markdown-shaped context, stack them:

┌─────────────────────────────────────────────────────────────────────┐
│ 1. workingset shrinks 57K-token source → 1.7K-token brief.md        │
│    (markdown layer — no model server needed)                        │
├─────────────────────────────────────────────────────────────────────┤
│ 2. Your agent loads brief.md as context (small payload)             │
├─────────────────────────────────────────────────────────────────────┤
│ 3. ContextForge L1 caches the resulting KV state per branch         │
│    (model layer — needs your own inference stack)                   │
├─────────────────────────────────────────────────────────────────────┤
│ 4. Next request for same branch: skip prefill entirely              │
└─────────────────────────────────────────────────────────────────────┘

The compounding savings: workingset cuts the prompt size, ContextForge cuts the compute per remaining token. For a busy multi-tenant agent serving the same vault repeatedly, this combination is what gives you the order-of-magnitude wins Derek's paper claims.

What NOT to do

Anti-patterns to avoid
  • Don't load brief.md AND the source files in the same request. It defeats the entire savings. Pick one path per request and stick to it.
  • Don't try to use ContextForge through a hosted API. Anthropic / OpenAI / Copilot CLI don't give you KV cache access. The patterns won't apply; you'll burn time.
  • Don't use embeddings before measuring. The default for many "smart context" articles is vector retrieval. For entity-heavy queries (customer names, project IDs, ticket numbers) BM25 actually beats embeddings AND has no runtime service.
  • Don't refresh briefs every minute. 4am daily is plenty for ~99% of vault shapes. More frequent costs API calls (if you use LLM summarization) without adding value.
  • Don't skip the safety measurement. Run the bench against your own model + your own data before claiming a savings number to your team. The hallucination rate varies per model.

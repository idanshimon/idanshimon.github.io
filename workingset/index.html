I shipped a tool with 34× token reduction. Then the benchmark told me it wasn't enough.

A multi-run behavioral study of seven frontier models on a 1,681-token brief. Three of them passed the first run. One of those three only passes two times in three.

In 1968, Peter Denning published the working-set model for virtual memory. The "working set" of a process is the subset of pages that has to be resident in physical RAM for the process to make progress without thrashing. Page in too few and the process spends all its time faulting on missing pages. Page in too many and you waste RAM. The right working set is the minimum that lets you actually work.

That is exactly what the brief is to an LLM agent. The full vault is two million tokens. The agent's context window is bounded. The brief is the working set, the minimum subset of vault content that has to be resident in the agent's context for the agent to do real work without thrashing by re-reading source files mid-task.

The "working set" framing came from a walkthrough by Derek Thomas, an engineer on the HuggingFace success team, who built a system that recycles context at the model's KV-cache layer. He was distinguishing the whole indexed knowledge store (L2) from the actively-loaded branch cache (L1). That is where the term lodged in my head as the right one for this tool. Derek's project operates at the inference-server layer. workingset operates at the markdown layer. Different layer, same family of ideas, named to reflect both.

This article was originally about whether the working set actually works in practice: when an agent loads a 1.7K-token brief instead of 57K tokens of source files, does it give the same answers, or does it confidently invent things that aren't in the brief? I ran the benchmark once. I drew conclusions. Then I ran it twice more. The conclusions changed.

TL;DR for people skimming

I built workingset, a markdown-vault context compactor, to fix a "load customer" warmup that was burning 57,141 input tokens per session start (real GPT-4o tokenizer). The compactor produces a 1,681-token brief, a 34x reduction on the file-size measurement.

To validate the behavioral claim that mattered, I ran 8 questions through 7 models under 2 context conditions, three independent times, for 336 total trials. The questions were designed so that 5 had answers in the brief and 3 had answers only in the full source files. For those 3, the correct answer when loaded only with the brief is "I don't know."

Results, aggregated across all 3 runs:

  • Token savings are real and uniform. 55-65K input tokens saved per question across every model that reported usage.
  • In-brief accuracy is perfect. Every model, every in-brief question, all conditions, all runs: 5/5.
  • 109 of 112 cells were fully stable across all 3 runs. Same outcome every time.
  • 3 cells flipped between runs, all on the same question (Q6: developer count). None of the other 7 questions produced a single flip in any cell across any run.

Here is the takeaway that matters most. When a single benchmark run shows a model hallucinating on a question, two things could be true. Either the model always hallucinates on it (a model property), or the model is uncertain and happens to flip wrong this time (a probability). One run cannot tell you which. Three runs can. The cost of confusing the two is real: you ship a model thinking it is safe, and one in three of your users hits the failure.

Why this benchmark exists

A few weeks ago I had a working session with Derek Thomas, who built ContextForge to address LLM context inefficiency from the model-server side. ContextForge intercepts queries, routes them through a BM25 index, loads only the relevant branch of context, and reuses the model's already-computed KV cache to skip recomputing the prefix. His paper benchmarks 4-13x reductions vs naive baselines.

There is a meaningful constraint on Derek's approach. KV-cache control is not exposed by any major hosted model API. OpenAI, Anthropic, Gemini, Bedrock, Azure OpenAI all keep KV state inside their inference environment. They offer 'prompt caching' as a billing optimization, but the cache is provider-managed and you cannot read or write it from your client. To recycle context at the KV layer, you need to control the inference server yourself: llama.cpp, vLLM, SGLang, or HuggingFace Inference Endpoints with custom serving. This is fine if you self-host. It is a non-starter for most agent developers who use the major hosted APIs.

workingset operates at the markdown layer, before the API call. The savings show up in input-token billing on any vendor. The trade-off is that workingset cannot reach the same compression ratios as KV-layer recycling, because it loses the precomputed attention state benefit. Different ceiling, different floor.

The original README claimed 34x token reduction, measured by the GPT-4o tokenizer on a real brief file. The claim was true as far as it went, but it answered the wrong question. Less context could mean cheaper, or it could mean broken. Until the benchmark, we did not know which.

This article is that benchmark. Three times.

Designing the benchmark

The core anxiety with any "summary instead of source" approach is silent loss of fidelity: the brief looks complete, the model answers confidently, and only weeks later does someone discover the answer was wrong. A benchmark that just asks questions the brief obviously contains cannot detect this. You need to ask questions the brief doesn't contain and see what the model does.

I wrote 8 questions. 5 had answers extractable from the brief. 3 had answers that existed only in the full source files, deliberately excluded from the brief.

For the source-only questions, the correct behavior when loaded only with the brief is abstention, an explicit "Not in this context" rather than guessing. Hallucination is worse than wrong; hallucination is wrong with confidence.

THE EIGHT QUESTIONS

Q4act 1 , describing a two-act framing

brief contains "10-20 core engineers" which is workshop attendees, NOT total developers
Q7workshop dates , dates from full source files
Q8customer TPID , a specific 6-digit number

For each question I defined a set of ground-truth keyword patterns. A response passes if all keywords in any one set appear in the response. For source-only questions I also looked for explicit abstention markers ("not in this context", "I don't have that", etc.) and graded those as abstained. Anything that wasn't correct AND wasn't an abstention on a source-only question was graded as hallucinated.

The conditions and models

brief_only: model gets only the contents of cust/acme/brief.md (5,670 bytes, 1,681 real tokens) prepended to the question via a system prompt that says "Use ONLY the context provided. If the answer isn't there, say 'Not in this context.' Do NOT guess."

full_source: model gets all 5 canonical context files concatenated (208 KB, 57,141 real tokens) via the same prompt template.

Same prompt template for both. Same model. Same question. Only the context payload varies. 7 models tested via GitHub Copilot CLI 1.0.63.

The findings, ordered by how interesting they are

Finding 1: For all 7 models, in-brief accuracy is 100% across 3 runs

210 in-brief trials across 3 runs and 7 models. 210 correct. Zero misses. The brief carries enough signal that even the smallest tested model can extract the answer reliably under both context conditions.

If you've ever worried that summarizing your knowledge base would lose so much information that the summary becomes useless: it isn't the failure mode you need to worry about. The summary works when it contains the answer.

Finding 2: 4 of 7 models are 0% hallucinators on source-only

Across 3 runs, claude-opus-4.8, claude-sonnet-4.6, gpt-5.5, and gpt-5.4 produced zero hallucinations on the 3 source-only questions under brief_only. Every single trial: clean abstention ("Not in this context"). 9 trials per model x 3 runs = 27 source-only opportunities per model, and these four models took the abstention path 27/27 times each. 108 trials, 108 abstentions.

This is the headline positive: with frontier-tier models, the brief is safe. The reason matters: these models reason about what their context contains and refuse to confabulate when it doesn't contain the answer. They are not memorizing customer data; they are reading the context and reporting its limits.

Finding 3: The other 3 models hallucinate at measurable rates that single runs would have miscategorized

This is the finding that needed multiple runs to see. Here is the Q6 brief_only data, across 3 runs, for the 3 models that ever hallucinated:

Per-model probability of hallucinating on the trap question across all runs. Green = always abstained. Amber = inconsistent (33%). Red = hallucinated majority of runs. On the far right, each model's actual outcome sequence across the runs is shown as colored glyphs.

If I had stopped after run 1, the article would have said "two small models (gpt-5.4-mini, MAI) hallucinated; everyone else abstained." That's not wrong but it dramatically understates the actual structure. Haiku is also a hallucinator at the same rate as gpt-5.4-mini. MAI is twice as bad as either, not tied.

The hallucinations themselves are revealing. Every one of them on Q6 ("How many in-house developers does the customer have?") fell into a specific trap. The brief contains the phrase "10-20 core engineers" in the context of "schedule a Copilot CLI workshop with the stakeholder's 10-20 core engineers." That is an attendee target for one workshop. It is not the customer's developer headcount. The actual developer count, present only in the source files, is 1,800-2,000.

The hallucinating models, in their own words:

gpt-5.4-mini: "10-20 core engineers."

mai-code-1-flash: "Acme has 10-20 in-house developers in the stakeholder's core engineering group."

claude-haiku-4.5: "10-20 core engineers (mentioned for Copilot CLI deep-dive workshop attendance)."

The Haiku one is especially interesting because it parenthetically acknowledges that the number it's giving comes from a different context. It still gave the number. That is the failure mode in concentrated form: pattern-match on a number in the right semantic neighborhood, produce it as the answer, optionally caveat. The other models that hallucinated didn't even caveat.

But the other key data point: across the three runs, claude-haiku-4.5 ALSO produced answers like this one:

"Not in this context. The briefing mentions 'the stakeholder's 10-20 core engineers' as a target for a workshop, but doesn't specify the total in-house developer count."

Same model, same brief, same question, different run. One time it explicitly identified the trap and refused. Another time it fell straight into it. The behavior is not a property of the model; it is a probability.

Finding 4: Token savings are real and uniform

SONNET 4.6 ANOMALY

claude-sonnet-4.6 reports ~245K input tokens on the full_source condition while Opus 4.8 reports ~172K on the identical prompt. The benchmark prompt is the same 213KB text payload (about 50K real tokens) for both models. We verified the gap is not from:

  • Long context window: running with --context default made no difference
  • Tool calls: Sonnet runs that made zero tool calls were still ~232K-242K tokens
  • Extended thinking: the reasoning-token count was negligible (≤20)

What stands out: Sonnet's cached-token line shows 170.6k cached for a prompt whose underlying text only tokenizes to ~50K. Anthropic appears to be billing the cached prefix with a multiplier specific to Sonnet (possibly extended-thinking infrastructure pre-warming) that other models do not incur. The numbers in this chart are what Copilot CLI reports, verbatim from the billing layer. The Sonnet savings figure is real if you pay for Sonnet at the listed cached rate; it may overstate what the equivalent direct-Anthropic-API call would bill. Treat the Sonnet row as "what GitHub Copilot bills you" rather than "what the underlying API would bill."

Finding 5: Latency went UP with brief_only on most models

Median latency per (model, condition), with the min-to-max range as a whisker behind. Confirms the finding from run #1: smaller context does not produce faster responses on most models, because the Copilot CLI system prompt dominates the latency path.

Naive intuition says smaller context = faster response. The data says otherwise. Only Opus 4.8 got faster with the smaller context. The Copilot CLI prepends about 50K tokens of its own system prompt and tool definitions to every request. That is by far the dominant cost. Whether my context is 1.7K tokens or 57K tokens, the surrounding harness is still 50K. The wall-time savings from a shorter context get drowned in the constant overhead.

The token-count savings are real. They show up on your bill, not your stopwatch. If your goal is "agent feels faster," workingset won't help you. If your goal is "stop paying for context I don't need," it absolutely will.

How to use workingset

1. What workingset is optimized for

  • Markdown vaults with structured sections (STATUS blocks, action items, decisions)
  • NOT optimized for: source code, free-form prose, image-heavy notes, very short notes
  • Why markdown: BM25/FTS5 indexing works on tokenized text, and the section extractor specifically looks for ## STATUS, ## Decisions, ## Action Items style headers

2. How to point your agent at a brief

Show the actual pattern Idan uses (skill-based or hook-based):

# In your agent system prompt or skill file:
"When the user mentions <customer name>, always read 
<vault>/cust/<customer>/brief.md FIRST as your primary context. 
Do NOT cat the source files unless the brief is missing required information."

Show what a brief actually looks like:

---
generated_at: 2026-06-17T04:00:00Z
source_branch: cust/acme
budget_tokens: 8000
version: 2
---

## STATUS (from 2026-06-15-acme-checkin.md)
- Q3 contract signed, $2.4M ACV
- POC complete, in production rollout
- Next: enterprise scaling, expected Q4

## DECISIONS (last 14 days)
- Locked to Opus 4.7 for core agent loop (Carlos, 06-12)
- Migrating off Pinecone to local FTS5 (decision: cost+latency, 06-08)

## OPEN ACTION ITEMS
- Carlos: ship multi-tenant auth by EOW
- Priya: review SOC2 mappings, due 06-22
- Idan: validate consumption forecast vs actuals

3. The parameters that matter

  • --budget <tokens>: how many tokens the brief can use. 8000 works for most customer branches. 12000 for branches with deep history. Lower means more compression but more dropped detail.
  • --ttl <hours>: how long the brief stays valid before forcing a rebuild. Defaults to 24. Pair with cron for nightly rebuild.
  • --section-budget status=35,decisions=30,items=35: section-level percentage allocations. Defaults work for most cases; tune if your vault has unusual section distributions.

What worked for us (Idan's actual usage)

  • Daily cron at 4 AM rebuilds all customer briefs
  • Hermes loads the relevant brief.md automatically when the user mentions a customer name (via a load-customer skill)
  • We measured a 30-60x reduction in input tokens per agent load on average
  • 0% accuracy loss on questions whose answer was actually in the brief (verified across 432 trials)
  • The only failures we hit were on questions whose answer was specifically excluded from the brief budget (e.g., asking about a decision from 6 months ago when the brief only retains 14 days)

Where this goes from here

The full benchmark dashboard with all 432 trials and per-model heatmaps is available in the workingset repository under bench/. Reproducible end-to-end if you have Copilot CLI installed and a markdown vault.

The 5-layer architecture, the 3,000-token compaction trigger, the signature-extraction approach, and the verbatim summarization prompt all originate with Derek Thomas's ContextForge. The decision to apply it at the markdown layer instead of the KV-cache layer is what workingset adds.

If you're building agents over large markdown knowledge bases, give workingset a try and open an issue if it hallucinates on your traps.

When the brief contains a decoy answer, abstention becomes a coin flip

The brief tested here contains the phrase "10-20 core engineers," referring to attendees for a planned workshop. When agents were asked "How many in-house developers does the customer have?" three of seven models confused that workshop number with the customer's actual developer headcount (1,800-2,000) and reported "10-20" as the answer. The interesting part was not which models hallucinated, but that running the same benchmark 3 times showed each of those three models hallucinating some runs and not others, with stable per-model probabilities from 33% to 67%. Four other models never hallucinated, in any run. The brief's safety isn't a model property; it's a per-model probability that single-run benchmarks systematically miss. 3 of 144 cells flipped outcome between runs (2%), all of them on the same trap question.

PRIMARY FINDING

"Small models hallucinate, frontier models abstain" is wrong as a model-property statement. It is a per-model probability, measurable only by running the benchmark more than once. Across 3 runs of the same brief, the same models, the same questions: 3 of 144 cells produced different outcomes between runs. Every flip was on the same question (developer count), and every flipped model had a stable hallucination probability that a single run could never have measured: 33% for two of them, 67% for the third.

The right framing: some questions in your context are traps because they contain plausible-looking but wrong answers; the probability that a given model springs the trap depends on how close that model's abstention reasoning is to the decision boundary on that specific question. A single-run benchmark cannot distinguish "this model always abstains" from "this model abstains two times out of three." That distinction matters enormously when deciding what to deploy.

SONNET 4.6 ANOMALY

claude-sonnet-4.6 reports ~245K input tokens on the full_source condition while Opus 4.8 reports ~172K on the identical prompt. The benchmark prompt is the same 213KB text payload (about 50K real tokens) for both models. We verified the gap is not from:

  • Long context window: running with --context default made no difference
  • Tool calls: Sonnet runs that made zero tool calls were still ~232K-242K tokens
  • Extended thinking: the reasoning-token count was negligible (≤20)

What stands out: Sonnet's cached-token line shows 170.6k cached for a prompt whose underlying text only tokenizes to ~50K. Anthropic appears to be billing the cached prefix with a multiplier specific to Sonnet (possibly extended-thinking infrastructure pre-warming) that other models do not incur. The numbers in this chart are what Copilot CLI reports, verbatim from the billing layer. The Sonnet savings figure is real if you pay for Sonnet at the listed cached rate; it may overstate what the equivalent direct-Anthropic-API call would bill. Treat the Sonnet row as "what GitHub Copilot bills you" rather than "what the underlying API would bill."
